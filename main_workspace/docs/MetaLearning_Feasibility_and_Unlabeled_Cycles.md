# Meta-Learning 可行性与无 LH 标签周期分析

**目的**：基于当前实验记录与数据集，分析 (1) 为何部分周期没有 LH 标签；(2) 排卵→月经的 pipeline 是否真的需要 personal 适配；(3) Meta-learning 是否可行、是否值得做。

---

## 一、哪些周期没有 LH 标签？为什么？

### 1.1 数据流与数量

```
cycle_clean_2.csv: 173 周期, 42 人
    ↓ get_lh_ovulation_labels() [原始]：ov_prob>0.5 + luteal∈[8,20]
95 个有标签，78 个被丢弃

    ↓ get_enhanced_ovulation_labels() [增强]：四级恢复策略 + 严格黄体期 [10,16]
111 个有标签，62 个仍无法恢复
```

### 1.2 无标签的 78 个周期（原始）— 分类

| 类别 | 数量 | 原因 |
|------|------|------|
| **ov_prob > 0.5 但 luteal ∉ [8,20]** | 33 | 黄体期过短(<8d)或过长(>20d)：生理异常或周期界定问题，不适合作为“标准排卵周期” |
| **ov_prob = 0（无排卵概率）** | 38 | LH surge 检测算法未捕获到明显 surge |
| **周期过短 (< 10 天)** | 2 | 数据不足，无法可靠标注 |
| **其他** | 5 | 与上重叠或边界 |

对 38 个「无概率」周期的细分（Enhanced 报告）：

| 子类 | 数量 | 含义 |
|------|------|------|
| Type A | 9 | 有明确 LH surge，但算法未检测到（可部分用 lh_peak+1 等恢复） |
| Type B | 11 | 无明显 LH surge（LH≤10），仅有 phase 标签，证据弱 |
| Type C | 7 | 仅有 Fertility 无 Luteal，周期被截断、不完整 |
| Type D | 11 | 无 Fertility 标签，无排卵证据 |

### 1.3 增强后仍无法恢复的 62 个周期

| 原因 | 数量 | 说明 |
|------|------|------|
| ov_prob>0.5 但 luteal ∉ [8,20] | 33 | 黄体期极端，多数过短，不宜作标准标签 |
| 无 phase 证据（无 Fertility/Luteal） | 11 | 无排卵阶段标注，无法推断排卵日 |
| Fertility 但无 Luteal（周期截断） | 7 | 周期未完整记录到下次月经 |
| 有 phase 但 luteal ∉ [10,16] | 9 | 恢复策略为保质量收紧到 [10,16]，这些被排除 |
| 周期 < 10 天 | 2 | 太短 |

**结论**：无 LH 标签主要不是因为“缺数据”，而是  
- **生理/界定**：黄体期异常、周期不完整、无排卵证据；  
- **算法**：部分有 surge 但未达到 ov_prob>0.5（增强后恢复了 16 个）。  
剩下 62 个要么无法可靠定义排卵日，要么不适合作为“标准周期”参与训练/评估。

## 二、当前「排卵→月经」pipeline 是否需要 personal 适配？

### 2.1 现有“个人化”做了什么

- **Personal luteal（两种来源）**  
  - **来自 LH 的 personal luteal**（`compute_personal_luteal_from_lh`）：用**金标准排卵日**算每人历史黄体期均值，仅用于分析/Oracle，**部署时不可用**。  
  - **来自检测的 personal luteal**（无 LH 的 pipeline）：用**历史“检测到的排卵日”+ 历史周期长度**估计每人平均黄体期（e.g. `past_cycle_len - past_detected_ov`），部署时可复现。

- **排卵前**：当前多用**全局模型**（如 LightGBM）+ 日历先验（hist_cycle_len、day_in_cycle 等），没有 per-user 模型参数适配。

### 2.2 实验结论（来自 Unsupervised_Ovulation_Menses_Report 与 Advanced 报告）

- **Oracle（排卵已知）时**：  
  - 固定黄体期 13d：±2d ≈ 81–84%，±3d ≈ 91%。  
  - 个人黄体期（来自 LH）：±2d 约 77%（**比固定 13d 略差**），±3d 约 93%（略优）。  
  即：在“完美排卵”下，personal 对 ±2d 无增益，对 ±3d 有轻微增益。

- **用检测到的排卵（非 Oracle）时**：  
  - 文档明确写：“**个人黄体估计因检测误差传播反而更差：固定 12–13 天优于自适应**”。  
  原因：personal luteal 来自「历史检测排卵日 + 历史周期长」，检测误差会进入黄体期估计，放大误差；固定 12–13d 反而更稳。

- **每人周期数**：中位数约 **3 个完整周期**（边界周期移除后），个人黄体期估计本身方差大，收益有限。

**结论**：  
- 在**排卵→月经**这一段，当前设计下 **不需要**、也**不适合**强做“个人适配”：固定黄体期 12–13d 已接近 Oracle 上限，且比“用检测估计的 personal luteal”更稳。  
- 真正限制整体表现的是**排卵检测精度**，而不是黄体期是否个人化。

### 2.3 代码复查：固定为何更稳、是否有实现问题（2026-03 复查）

**实现逻辑核对**（`run_final_ov_menses.py` / `run_final_ov_menses_v2.py`）：

- **personal 来源**：在「检测排卵 + personal」设定下，`subj_past_luteal[uid]` 在**按时间顺序**遍历每人周期时，用**已处理过的历史周期**更新：  
  `el = actual - det[0]`（当前周期真实长度 − 当前周期检测排卵日），仅在 `8 <= el <= 20` 时加入（与 `get_lh_ovulation_labels` 一致）。  
  预测当前周期时只用 **past_luts**（之前周期的列表），当前周期处理完才把当前 `el` 加入，**无泄漏**。

- **为何固定更稳（无 bug 的合理解释）**：  
  1. **检测误差传播**：`el = actual - detected_ov`。若检测系统性偏晚 2 天，则 `el` 系统性偏小 2 天，personal 均值偏小，下一周期预测会偏早；固定 12–13d 不受单周期检测误差影响。  
  2. **样本量小**：多数用户只有 1–2 个**有标签**历史周期，personal = mean(1～2 个数)，方差大；固定 13d 是总体均值，更稳。

- **Oracle + personal 在同一脚本中**：Oracle 时 `det[0]` 为真实排卵日，故 `el` 为真实黄体期；personal 仍只来自**已处理过的历史周期**，无泄漏。Oracle + personal 有时比固定差，主因是**历史仅 1 个周期时** personal 为单点，易受该周期黄体期波动影响；固定 13d 更稳健。

- **与 LH 标签范围一致**：已在 `run_final_ov_menses.py` / `run_final_ov_menses_v2.py` 中将 past_luteal 过滤改为 `8 <= el <= 20`，与 `get_lh_ovulation_labels` 一致。

**其他脚本中的潜在泄漏（已修复）**：  
`run_oracle_experiment.py` 与 `run_detected_ov_experiment.py` 原先使用全局 `compute_personal_luteal_from_lh()`，在 **subject-level 划分** 下对测试用户会用到该用户所有周期（含当前被预测周期）的 LH 黄体期，存在泄漏。已改为：仅对**训练集用户**使用 personal_luteal，测试用户统一用总体均值 `pop_luteal_mean`。

## 三、Meta-Learning 可行性

### 3.1 数据规模（与 meta-learning 的匹配）

- **人数**：40（边界移除后）或 42；有标签周期 111，增强后。  
- **每人周期数**：中位数约 **3**，范围约 1–8（173 周期时）；边界移除后不少人只有 **1–2 个完整周期**。  
- **任务定义**：若把“每个用户”看成一个 task，则  
  - support set：该用户部分周期（e.g. 1–2 个周期）；  
  - query set：该用户另一周期。

### 3.2 Meta-learning 的典型需求

- **Task 数量**：通常希望几十到上百个 task（用户），用于 outer-loop 更新。  
  → 40 个用户勉强够用，但偏少。  
- **每 task 的 support 规模**：MAML 等通常希望每 task 有 **5+** 样本（这里一个“样本”若为周期级别，即 5+ 周期/人）。  
  → 我们中位数只有 **3 周期/人**，且很多只有 1–2 个，support 非常小，inner-loop 容易过拟合或方差极大。  
- **信号**：排卵检测/月经预测依赖的腕温等信噪比低（文档中 SNR 约 0.32），**任务本身噪声大**，meta-learning 容易学到噪声而非可迁移结构。

### 3.3 与当前 pipeline 的契合度

- 当前**主要瓶颈**是：  
  - 排卵检测不准（可穿戴信号限制）；  
  - 排卵后用固定黄体期已经接近理论上限。  
- Meta-learning 解决的是“**少样本快速适配新用户**”。但：  
  - 你这边“个人化”在排卵后段**收益有限**（见上），且 personal luteal 用检测估计会变差；  
  - 若把 meta-learning 用在**排卵检测**上，每用户 1–2 个周期做 support 几乎不够学出稳定用户特定参数；  
  - 若用在**排卵前预测（日历+信号）**，同样面临每用户样本极少、且信号弱的问题。

### 3.4 结论：可行性 vs 是否值得

| 维度 | 结论 |
|------|------|
| **可行性** | 技术上可以跑 MAML 等（e.g. 周期级或日级样本），但 **support 极小**（1–3 周期/人）、**task 数有限**（40），结果容易不稳定、难以得出可靠结论。 |
| **是否值得** | **当前数据与 pipeline 下不推荐**：  
  1) 排卵→月经段个人适配收益小，固定黄体期已够用；  
  2) 每用户周期数太少，meta-learning 的“快速适配”优势难以体现；  
  3) 一个月内要出结果+写论文，meta-learning 实现与调参成本高、收益不确定。 |

若仍想尝试“个人化”方向，更现实的是：  
- **轻量个人化**：例如仅用**每人历史 1–2 个周期**估计一个常数偏移或个人黄体期，与全局模型组合（你已有 personal luteal 实验，结论是固定更稳）；或  
- 把有限时间放在** Conformal prediction、可预测性分层、或半监督利用无标签周期**等与当前瓶颈更相关的方向上。

---

## 四、简短总结

1. **无 LH 标签的周期**：多数因黄体期异常、周期不完整或无排卵证据，无法可靠定义排卵日；增强后仍剩 62 个无法恢复，不宜强行当“可标注”使用。  
2. **Personal 适配**：排卵→月经用固定黄体期 12–13d 已接近上限；用检测估计的 personal luteal 会因误差传播而变差，当前 pipeline **不依赖、也不建议**强做个人适配。  
3. **Meta-learning**：数据上每用户周期太少、任务噪声大，**可行性有限、性价比低**；更建议把精力放在方法上更贴合当前瓶颈的改进（如区间预测、可解释的可信度、或半监督）。
